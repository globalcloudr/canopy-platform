# Canopy Designer Handoff Brief

Repository: `/Users/zylstra/Code/canopy-platform`  
Purpose: Give a designer a clear, practical starting point for Canopy visual design work  
Priority: High  
Context: Canopy needs to move to market quickly without skipping strategic and UX clarity

## Why This Brief Exists

Canopy is still in the product-structure and MVP-definition stage.

That means the current portal work is a concept mock, not final design.

The goal of this brief is to help a designer start from:

- a clear platform concept
- a clear dashboard model
- a clear product and service structure
- a realistic MVP scope

Instead of starting from:

- scattered notes
- unfinished prototypes
- architecture docs without a visual brief

## Product Summary

Canopy is the parent platform.

It is a connected school growth platform that brings together products and services for:

- assets
- newsletters
- social posting
- websites
- communications
- visibility

Canopy is not centered on PhotoVault.

PhotoVault is one product within the Canopy platform, alongside:

- `Community Canopy`
- `Reach Canopy`
- `Canopy Web`
- `Create Canopy`
- `Publish Canopy`
- `Assist Canopy`
- `Insights Canopy`

## Audience

Primary users:

- adult school administrators
- school staff
- communications and marketing staff
- front office users

Secondary users later:

- multi-site school operators
- Canopy internal staff
- district-style users or multi-organization users

## Product Positioning

Canopy should feel:

- practical
- modern
- clear
- approachable
- organized
- school-friendly
- operational

Canopy should not feel:

- luxury
- boutique
- fashion-brand polished but impractical
- enterprise-bloated
- overly technical
- abstract

Internal shorthand:

- more `Walmart` than `Prada`
- broad, useful, familiar, and accessible

This does not mean ugly or cheap.

It means:

- useful first
- easy to understand
- broad appeal
- efficient daily use

## Market Context

Directionally, Canopy competes in the same general space as:

- Finalsite
- Edlio
- Apptegy
- ParentSquare
- SchoolStatus
- CatapultCMS

Useful design takeaway:

- public Canopy experience can learn from the polish and confidence of Finalsite
- authenticated Canopy portal should learn more from Edlio's dashboard practicality and role-aware usability

## Core UX Principle

The portal should follow normal SaaS expectations.

Standard flow:

1. user visits homepage
2. user clicks `Sign in`
3. user lands on dashboard
4. user sees what their organization has access to
5. user clicks the task they came to do

The portal should not begin with:

- workspace confusion
- technical choices
- architecture concepts
- generic product-launch behavior

## MVP Product Sequence

The first Canopy MVP should feel real around three workflows:

1. `PhotoVault`
   - first connected product
   - photo and asset management

2. `Community Canopy`
   - second connected product
   - create and send newsletters

3. `Reach Canopy`
   - third connected product
   - create and schedule social posts

## Dashboard Concept

The dashboard is the center of the authenticated Canopy experience.

It should answer:

- what does this organization have?
- what can this user do today?
- what is ready to use now?
- what other Canopy offerings are available?
- what services are being provided?

## Dashboard Sections

Recommended MVP dashboard structure:

### 1. Welcome / Context

Shows:

- organization name
- signed-in user
- short orientation copy

Should feel:

- calm
- useful
- not over-explained

### 2. Your Apps

This section shows:

- the products the organization currently has access to
- products already enabled
- products actively in setup or pilot if relevant

Examples:

- `PhotoVault`
- `Community Canopy`
- `Canopy Web` if in setup

### 3. Other Products

This section shows:

- additional Canopy products not currently enabled
- these should feel visible but secondary

Examples:

- `Reach Canopy`
- `Insights Canopy`
- `Create Canopy`

### 4. Services

This section shows:

- managed offerings
- support or implementation services
- not all useful things need to be standalone apps

Examples:

- `School Website Setup`
- `Design Support`
- `Communications Support`

### 5. Account / Support Entry

Not necessarily a main hero area, but the system should make room for:

- account details
- support contact
- request-access flows

## Card Behavior

Each product or service card should support:

- name
- short description
- status
- primary action
- optional secondary action

Product cards should be task-oriented.

Good examples:

- `View Photos`
- `Open Brand Portal`
- `Create Newsletter`
- `View Campaigns`
- `Edit Website`
- `Start Setup`

Avoid generic labels like:

- `Launch Product`

unless there is truly no more specific action available.

## Page Types To Design

The designer does not need to design every future screen first.

Priority page types:

1. Public homepage
2. Sign-in page
3. Authenticated dashboard
4. Product/service card system
5. Account/support entry point

Secondary later:

- request-access page
- product overview pages
- service request pages

## Visual Direction

Desired qualities:

- clean
- modern
- confident
- warm without being soft or vague
- readable
- calm for daily use
- strong hierarchy
- simple interaction cues

Portal-specific goals:

- dashboard should feel easy to scan
- cards should feel useful, not decorative
- navigation should be obvious
- users should feel oriented quickly

Homepage-specific goals:

- clearer storytelling
- stronger branding
- more expressive than the portal
- but still connected to the same platform family

## Design System Direction

Recommended shared system:

- one Canopy design system
- two expressions:
  - marketing/public
  - portal/product

The system should define:

- typography
- colors
- spacing
- cards
- buttons
- forms
- navigation
- page rhythm
- responsive behavior

The portal should not just be a shrunken marketing page.

It needs its own operational UI logic.

## Functional Constraints

The design should support:

- one-sign-in dashboard-first workflow
- products and services separated clearly
- multiple product states:
  - enabled
  - in setup
  - pilot
  - not enabled
  - service
- future multi-organization support without making that the default visible experience

## What The Designer Should Not Assume

The designer should not assume:

- PhotoVault is the platform center
- every Canopy product already exists in production
- the current portal mock is the intended final visual design
- all future products need identical interfaces

## Deliverables To Request From A Designer

Recommended first deliverables:

1. visual direction exploration
2. homepage concept
3. sign-in page concept
4. dashboard concept
5. product/service card system
6. lightweight component foundation

If time is limited, the highest-value deliverables are:

1. dashboard design
2. homepage design
3. shared component patterns

## Success Criteria

The design phase should produce a Canopy experience that:

- looks credible against larger competitors
- feels clear to school users
- supports quick MVP implementation
- does not confuse users about products versus services
- makes Canopy feel like one platform with multiple offerings
