Direction C · Design system charter

A system with a point of view.

Option A was a refresh. Option B was a redesign. Option C is a constitution. Below are the rules every Canopy product follows. A system is worth the effort only when the rules are opinionated enough to produce recognizable family resemblance — and loose enough that each product can do its actual job.

01Hierarchy

Type does the hierarchy.

Size, weight, and leading do the structural work. Borders, fills, and shadows are a last resort, not a first instinct.

You cannot add a card to fix a layout problem. If the content wouldn't read without one, the content is wrong.

No default cards44→24→16→13 scaleFraunces display · Inter UI
02Marquee

One big moment per screen.

Exactly one element is allowed to be visually loud: a greeting, a cover photo, a hero illustration. Everything else supports it.

Stacked heroes are banned. Two gradients on one page is banned. A dashboard with three navy gradient banners is the antithesis of this system.

One marquee, maxOthers earn attention by content
03Density

Density follows content.

Utility views (tables, lists, admin) are tight — 32–40px rows, minimal padding, edge-to-edge when it helps. Showcase views (dashboards, profiles, marketing) are airy — generous gutters, big type, real negative space.

Uniform padding across every surface is how AI-generated design systems telegraph themselves.

Per-context, not per-tokenEarn generosity
04Identity

One accent per surface. The spine stays Canopy.

Every product overrides exactly one variable — --accent — from Canopy's palette. Everything else (type, spacing, motion, layout primitives) inherits from the core.

Walk into any Canopy product: you should know it's Canopy in 200ms and which one in 400ms.

PhotoVault · coralStories · oceanReach · violetCreate · amberCanopy · olive-sage
05Chrome

Chrome is earned, not default.

An element gets a border or a fill — never both, except inputs. One shadow, ever, from one token. Radii come in three sizes only: tight (4px), soft (10px), large (20px, marquee only).

This is where the current Canopy system leaks AI-design-default: 28px radii on everything, double borders on cards, stacked shadows. Forbidden here.

Border OR fillOne shadow tokenThree radii
06Motion

One easing, one rhythm, applied everywhere.

All transitions use cubic-bezier(0.2, 0.8, 0.2, 1). Durations: quick (140ms), normal (220ms), slow (360ms). Nothing linear, ever.

Motion is the cheapest way for a B2B product to read as crafted — most competitors skip it entirely.

One ease curveThree durationsNo linear
07Voice

Fraunces for moments. Inter for work.

Serif for headings, greetings, quotes, brand moments. Sans for data, forms, tables, body copy. The italic cut of the display face is used sparingly — for names, one word per headline, and the single accent word in a phrase.

This is the single biggest lever for the "warm and human" target.

Fraunces displayInstrument Serif italicInter UI, ss01 + cv11
ArchitectureTwo token layers. One package. Five consumers.

Base tokens (Canopy core)

Defined once in packages/ui/src/styles.css. Every product, every repo inherits these. They do not change per product.

  • Typography (fonts, sizes, leading, tracking)
  • Spacing scale (4 / 8 / 12 / 16 / 24 / 36 / 56 / 88)
  • Radii (tight / soft / large)
  • Motion (ease, durations)
  • Neutral palette (canvas, ink, muted, faint, lines)
  • One shadow token

Product tokens

Each product overrides one class at the root of its app. That class changes only accent-related variables.

  • .product-photovault → coral accent
  • .product-stories → ocean accent
  • .product-reach → violet accent
  • .product-create → amber accent
  • .product-canopy → olive-sage (default)

Result: one npm package, five repos, five identities, zero drift.

RolloutHow this actually ships across five repos.
  1. Wire all products to @canopy/ui. Platform already publishes it; PhotoVault, Stories, Reach, and Create need to consume it (likely via workspace dep or private npm). Without this, every repo is its own snowflake forever.
  2. Freeze tokens in Canopy/ui, not per app. Today styles.css has Tailwind-default values. Replace with the Option C palette, typography, motion tokens. This is a PR, not a project.
  3. Add the product class. Each repo sets class="product-photovault" (or equivalent) on its body / root layout. One line per product.
  4. Commission or license the display face. Fraunces is open source and free to use; Instrument Serif too. Zero licensing blocker. If you want a fully custom face, that's a separate 6–10 week commission.
  5. Migrate surfaces in order of visibility. Marketing homepage and sign-in first (highest-impression surfaces). Portal dashboard next. Then per-product dashboards. Each product interior last, worked by whichever team owns that repo.
  6. Enforce with lint. Add a stylelint rule that rejects hardcoded colors, non-token radii, and non-system shadows. The principles get real only when they're automated.
  7. Write no new components during the migration. Rule of thumb: a component has to earn its way in by replacing at least three one-off implementations. Otherwise you're just rebuilding a different generic library.
